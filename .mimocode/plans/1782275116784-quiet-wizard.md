# Plan: Improve Merge Preset — Intelligent Sub-Task Routing + Sakana Fugu Integration

## Context

The current merge mode (`ResultAggregator` at `result-aggregator.ts:31-92`) is a single-LLM pass that concatenates sub-task outputs and asks one model to merge them. It has no intelligence about *which* model should handle *which* subtask.

**Research findings (Sakana Fugu + TRINITY + Conductor):**
- **Sakana Fugu** (Sakana AI, Japan) — dynamically orchestrates a pool of specialized LLMs via a learned coordinator. Fugu Ultra beats Opus 4.8, Gemini 3.1 Pro, and GPT 5.5 on SWE-Bench Pro (73.7), LiveCodeBench (93.2), TerminalBench (82.1).
- **TRINITY** (ICLR 2026, arXiv:2512.04695) — 0.6B coordinator + 10K head, evolved via CMA-ES. Assigns Thinker/Worker/Verifier roles across turns. 86.2% LiveCodeBench.
- **Conductor** (ICLR 2026, arXiv:2512.04388) — 7B model trained via RL. Learns communication topologies and prompt engineering strategies. Adapts to arbitrary agent pools.
- Key insight: **don't send all subtasks to the same model** — route each to the model best suited for its type (code gen → coding model, reasoning → reasoning model, analysis → analysis model).

## Goal

Transform merge mode from "one model merges N outputs" into a 3-phase pipeline:
1. **Decompose** the task into typed subtasks
2. **Route** each subtask to the best-matching model from a pool
3. **Synthesize** all results via a judge model with structured analysis

This makes merge a lightweight version of what Fugu does: task-adaptive model routing + synthesis.

## Implementation

### Step 1: Extend types (`coordinator/types.ts`)

Add new types for the routing system:

```ts
export type SubTaskType =
  | 'code_generation'
  | 'code_review'
  | 'reasoning'
  | 'analysis'
  | 'research'
  | 'summarization'
  | 'general';

export interface ModelCapability {
  modelId: string;
  specialties: SubTaskType[];    // what this model is good at
  tier: 'cheap' | 'mid' | 'frontier';
  costPerMillionInput: number;
  costPerMillionOutput: number;
}

export interface ModelPool {
  models: ModelCapability[];
  /** If true, always use frontier model for merge synthesis */
  preferFrontierForJudge?: boolean;
}

export interface MergeConfig {
  pool: ModelPool;
  /** Max subtasks to spawn in parallel */
  maxConcurrency?: number;       // default 4
  /** If true, run a judge synthesis pass over merged output */
  enableJudgeSynthesis?: boolean; // default true
  /** Budget cap in USD for the entire merge operation */
  budgetUsd?: number;
}

export interface SubTaskResult {
  subTaskId: string;
  status: 'success' | 'error' | 'timeout';
  output: string;
  tokensUsed: number;
  assignedModel?: string;        // which model handled it
  subTaskType?: SubTaskType;     // what type it was classified as
  error?: string;
  durationMs: number;
}
```

### Step 2: Rewrite `ResultAggregator` (`result-aggregator.ts`)

Replace the current single-pass merge with a 3-phase pipeline. The class keeps the same interface (`aggregate`) but gains internal routing logic.

**Phase 1 — Classify Subtasks:**
- Each `SubTaskResult` gets classified into a `SubTaskType` via keyword heuristics (fast, no LLM call):
  - `code_generation`: contains "implement", "write code", "create function", "build", "scaffold"
  - `code_review`: contains "review", "audit", "check", "security", "vulnerability"
  - `reasoning`: contains "analyze", "compare", "evaluate", "tradeoff", "why"
  - `research`: contains "research", "find", "investigate", "report"
  - `summarization`: contains "summarize", "condense", "overview", "executive"
  - `analysis`: contains "metrics", "benchmark", "measure", "quantify"
  - `general`: fallback

**Phase 2 — Route to Best Model:**
- For each subtask, select the model from the pool whose `specialties` best match the `SubTaskType`
- Tiebreak by tier (prefer cheapest that qualifies), then by cost
- If a subtask has no matching model, use the pool's cheapest model
- Re-execute subtasks that need a different model than originally assigned (or skip if already handled well enough — configurable)

**Phase 3 — Synthesize with Judge:**
- If `enableJudgeSynthesis` is true:
  - Pick the best model as judge (frontier tier if `preferFrontierForJudge`, otherwise highest-specialty-match)
  - Send all routed outputs to the judge with a structured merge prompt (enhanced version of current `MERGE_PROMPT`)
  - Judge produces: `mergedOutput`, `conflicts`, `consensus`, `uniqueInsights`, `blindSpots`, `confidence`
- If disabled or budget exceeded: fall back to deterministic merge (concatenation + simple dedup)

### Step 3: Update `CoordinatorEngine` (`coordinator-engine.ts`)

- Accept optional `MergeConfig` in constructor params
- Thread the config to `ResultAggregator`
- Add cost tracking: sum token costs across all subtask executions, check against `budgetUsd`
- If budget exceeded mid-pipeline, degrade gracefully (skip judge synthesis, use deterministic merge)

### Step 4: Enhanced Merge Prompt

Replace the current `MERGE_PROMPT` with a more structured version:

```
You are Chimera's Result Synthesis Engine.

Given outputs from specialized sub-agents (each handling a different aspect of the task),
produce a single authoritative response.

MANDATES:
1. CONFLICT RESOLUTION: Identify contradictions between outputs.
2. DEDUUP: Merge overlapping content into single sections.
3. GAP ANALYSIS: Identify what no sub-agent covered.
4. UNIQUENESS: Preserve unique insights from each sub-agent.
5. QUALITY: Rank conflicting approaches by merit, not recency.

OUTPUT (JSON):
{
  "mergedOutput": "unified authoritative response",
  "conflicts": [{ "subTaskIds": [], "type": "contradiction|overlap|gap", "description": "", "resolution": "" }],
  "consensus": ["point of agreement"],
  "uniqueInsights": ["insight only one sub-agent provided"],
  "blindSpots": ["topic no sub-agent addressed"],
  "confidence": 0.0-1.0,
  "resolved": boolean
}
```
