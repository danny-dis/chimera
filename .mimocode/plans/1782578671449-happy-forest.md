# Plan: Supercharge Chimera with Omnigent Learnings

## Executive Summary

Omnigent is a meta-harness that orchestrates multiple coding agents (Claude Code, Codex, Cursor, OpenCode, Pi, Hermes) through a unified interface. Chimera can adopt several proven patterns from Omnigent to expand its capabilities ‚Äî particularly around **agent harness integration**, **gateway/provider flexibility**, **policy governance**, **cloud sandbox execution**, and **multi-device session sync** ‚Äî while preserving Chimera's core identity as a terminal-native, parallel multi-agent coding platform.

## What to Adopt (High Value, Low Risk)

### 1. Agent YAML Specification ‚Äî Declarative Agent Definitions
**Source:** Omnigent's `agent.yaml` spec
**Target:** `packages/chimera-core/src/agent/agent-schema.ts` (new) + `packages/chimera-workflows/src/schemas/`

Omnigent agents are defined as short YAML files with `name`, `prompt`, `executor` (harness), and `tools` (function/mcp/agent types). Chimera already has a workflow engine with 7 DAG node variants. Adding an agent YAML spec would let users define reusable agent configurations that the orchestrator can spawn.

**Changes:**
- Create `packages/chimera-core/src/agent/agent-schema.ts` ‚Äî Zod schema for agent YAML definitions
- Add agent YAML loading to `packages/chimera-core/src/agent/` directory
- Wire agent definitions into the existing Task Router (`packages/chimera-core/src/task-router.ts`)
- Add `agent:` tool type to DAG node schema in `packages/chimera-workflows/src/schemas/dag-node.ts`

```yaml
# Example Chimera agent YAML (adopted from Omnigent pattern)
name: code-reviewer
prompt: |
  You are a code reviewer. Review diffs for correctness,
  security, and maintainability. Return structured findings.
executor:
  provider: anthropic
  model: claude-sonnet-4-20250514
tools:
  git_diff: inherit
  read_file: inherit
  linter:
    type: function
    callable: chimera-tools.linter.run
```

### 2. Gateway Provider Support ‚Äî OpenRouter, Ollama, Databricks
**Source:** Omnigent's gateway credential system
**Target:** `packages/chimera-providers/src/providers/` (new files) + `packages/chimera-providers/src/provider-registry.ts`

Omnigent supports OpenRouter, LiteLLM, Ollama, vLLM, Azure via `base_url` + key. Chimera's provider registry currently has only 17 lines. Adopt Omnigent's gateway pattern to support any OpenAI-compatible or Anthropic-compatible endpoint.

**Changes:**
- Create `packages/chimera-providers/src/providers/openrouter.ts` ‚Äî OpenRouter adapter
- Create `packages/chimera-providers/src/providers/ollama.ts` ‚Äî Ollama local adapter
- Enhance `packages/chimera-providers/src/provider-registry.ts` ‚Äî add `ProviderCapabilities` from Omnigent's typed pattern (tiered `structuredOutput: 'enforced' | 'best-effort' | false`)
- Create `packages/chimera-providers/src/gateway-adapter.ts` ‚Äî generic base_url + key adapter
- Update `packages/chimera-providers/src/types/` ‚Äî add `ProviderCapabilities` interface

### 3. Policy Governance ‚Äî Three-Level Policy Stacking
**Source:** Omnigent's policy system
**Target:** `packages/chimera-tools/src/permission/policy.ts` (enhance)

Omnigent policies stack across three levels: **server-wide** (admin), **per-agent** (developer), **per-session** (user), with stricter rules checked first. Chimera already has a `PermissionEngine` with profiles (readOnly/editFiles/fullAccess/custom). Extend it with:

**Changes:**
- Add policy stacking to `packages/chimera-tools/src/permission/policy.ts` ‚Äî `PolicyStack` class that merges server ‚Üí agent ‚Üí session policies
- Add `ask_on_os_tools` builtin policy ‚Äî ask before shell/file writes
- Add `max_tool_calls_per_session` builtin ‚Äî cap tool calls per session
- Add `cost_budget` builtin with `ask_thresholds` ‚Äî soft warnings at thresholds, hard stop at cap
- Create `packages/chimera-tools/src/permission/builtins.ts` ‚Äî builtin policy factory functions
- Wire policies into `packages/chimera-core/src/session-orchestrator.ts`

### 4. Harness Swapping ‚Äî Run Across Multiple Agent Backends
**Source:** Omnigent's multi-harness architecture
**Target:** `packages/chimera-core/src/agent/` (new harness abstraction)

Omnigent lets the same agent definition run across Claude Code, Codex, Cursor, OpenCode, Pi, Hermes. Chimera should adopt this as a **harness abstraction** ‚Äî letting users choose which agent backend to use for different roles.

**Changes:**
- Create `packages/chimera-core/src/agent/harness-registry.ts` ‚Äî registry of available harnesses
- Create `packages/chimera-core/src/agent/harnesses/` ‚Äî harness adapters:
  - `claude-code.ts` ‚Äî Claude Code SDK integration
  - `codex.ts` ‚Äî Codex CLI integration
  - `opencode.ts` ‚Äî OpenCode integration
- Add `harness` field to agent YAML schema
- Wire harness selection into Task Router for role-based assignment

### 5. Cloud Sandbox Execution ‚Äî Disposable Execution Environments
**Source:** Omnigent's managed host system
**Target:** `packages/chimera-isolation/src/providers/` (new files)

Omnigent runs agents in disposable sandboxes (Modal, Daytona, E2B, CoreWeave, Kubernetes). Chimera already plans worktree isolation. Add cloud sandbox providers.

**Changes:**
- Create `packages/chimera-isolation/src/providers/e2b.ts` ‚Äî E2B sandbox adapter
- Create `packages/chimera-isolation/src/providers/modal.ts` ‚Äî Modal sandbox adapter
- Create `packages/chimera-isolation/src/sandbox-registry.ts` ‚Äî registry of sandbox providers
- Enhance `packages/chimera-isolation/src/types.ts` ‚Äî add `SandboxProvider` interface
- Add `sandbox` field to config.yaml schema

## What to Adapt (High Value, Needs Reshaping)

### 6. Multi-Device Session Sync
**Source:** Omnigent's session-follows-you pattern
**Target:** `packages/chimera-session/` (enhance) + `packages/chimera-daemon/` (enhance)

Omnigent sessions sync across terminal, browser, phone. Chimera's session store already exists. Adapt the pattern:

**Changes:**
- Enhance `packages/chimera-session/src/session-store.ts` ‚Äî add event-stream based sync protocol
- Enhance `packages/chimera-daemon/` ‚Äî add WebSocket endpoint for real-time session sync
- Create `packages/chimera-session/src/sync-protocol.ts` ‚Äî delta-sync protocol for session state
- NOT full browser/phone UI ‚Äî just the sync backend that future UIs can consume

### 7. Real-Time Collaboration ‚Äî Session Sharing & Co-Driving
**Source:** Omnigent's collaboration features
**Target:** `packages/chimera-daemon/` (enhance)

**Changes:**
- Add session sharing protocol to daemon ‚Äî generate shareable session tokens
- Add co-drive mode ‚Äî multiple clients can send messages to the same session
- Add session forking ‚Äî clone a session and continue independently
- These leverage the existing JSON-RPC daemon, just add new methods

### 8. OIDC / Multi-User Authentication
**Source:** Omnigent's auth system
**Target:** `packages/chimera-daemon/` (new auth module)

**Changes:**
- Create `packages/chimera-daemon/src/auth/` ‚Äî auth module
- Support GitHub, Google OAuth for session sharing
- User-scoped config and API keys
- Optional ‚Äî only needed when collaboration features are used

## What to Skip (Low Value for Chimera)

| Omnigent Feature | Why Skip |
|---|---|
| Platform adapters (Slack/Telegram/Discord) | Chimera is TUI-first. Multi-platform dilutes the single-agent identity |
| Web UI / React frontend | Chimera is terminal-native. Skip unless explicitly requested |
| Multi-tenancy (per-user DB tables) | Chimera is single-user. Keep it simple |
| Dockerfile.user.example pattern | Only needed when Chimera ships a Docker image |
| Homebrew formula | Chimera is pnpm-based; npm homebrew is wrong shape |
| PostHog telemetry | Skip until Chimera has users who want analytics |

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Gateway adapter breaks with provider API changes | Medium | Medium | Pin provider SDK versions, add integration tests |
| Policy stacking complexity increases bug surface | Low | Medium | Keep policy stack simple ‚Äî merge + override, no nesting |
| Cloud sandbox providers require API keys we can't test | High | Low | Make sandbox providers optional, graceful fallback to worktree |
| Session sync introduces race conditions | Medium | High | Use event-sourcing pattern, idempotent operations |
| Harness abstraction adds latency | Low | Medium | Lazy-load harnesses, cache connections |

## Success Metrics

1. **Gateway support:** Can connect to OpenRouter, Ollama, and any OpenAI-compatible endpoint
2. **Agent YAML:** Can define agents in YAML and have the orchestrator spawn them
3. **Policy governance:** Three-level policy stacking works (server ‚Üí agent ‚Üí session)
4. **Cloud sandbox:** At least one cloud sandbox provider (E2B) works end-to-end
5. **Harness swapping:** Can run the same agent definition on Claude Code and Codex
6. **Session sync:** Can resume sessions across terminal restarts with full state
7. **Cost:** No regressions in existing cost tracking
8. **Tests:** All existing tests pass, new features have >80% coverage
