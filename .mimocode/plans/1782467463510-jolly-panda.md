# Streaming Response Display for Chimera VS Code Extension

## Problem

When a user sends a message in the Chimera chat panel, they see a spinner while waiting for the **entire** LLM response to complete. The response only appears once fully generated. Users should see output streaming in real-time — tokens appearing as they're generated.

## Current Flow

```
User input → chat-panel.processUserInput() → daemon.call('execute_task') 
→ [blocks until complete] → result.output → addMessage('assistant', output) → render
```

## Proposed Streaming Flow

```
User input → chat-panel.processUserInput() 
→ daemon.call('stream_task') [returns immediately with taskId]
→ daemon subscribes to EventStream for that task
→ daemon sends 'token' notifications over JSON-RPC 
→ daemon-client routes to onEvent handler
→ chat-panel appends tokens to the assistant message in real-time
→ on 'final_response' event, finalize the message
```

---

## Implementation Plan

### Step 1: Add `token_stream` event type to event schema

**File:** `chimera/packages/chimera-core/src/types/events.ts`

Add a new discriminated union variant:
```typescript
z.object({
  type: z.literal('token_stream'),
  taskId: z.string(),
  content: z.string(),        // incremental token content
  finishReason: z.string().optional(),
  isFinal: z.boolean(),
})
```

### Step 2: Add `stream_task` JSON-RPC method to daemon

**File:** `chimera/packages/chimera-daemon/src/server.ts`

Add a new `stream_task` method that:
1. Creates the worker and starts `executeTask` in the background (non-blocking)
2. Subscribes to the EventStream for `token_stream` events matching the taskId
3. Returns immediately with `{ taskId, status: 'started' }`
4. As the background task runs, forwards `token_stream` and `final_response` events as JSON-RPC notifications to stdout

The key insight: we don't need to modify the SessionOrchestrator's core execution. Instead, we use a **polling bridge** — after `executeTask` completes, we emit a `final_response` event with the full output. For a minimal viable streaming experience, we stream the intermediate events (tool calls, agent spawns, draft proposals, etc.) and then the final response. True token-level streaming requires modifying the orchestrator to use `provider.stream()` instead of `provider.complete()`, which is a larger change.

**Minimal streaming approach (recommended for 1-3 day scope):**

Instead of true token-level streaming, implement **event-level streaming** — the daemon subscribes to the EventStream and forwards all events as they occur. This gives users real-time visibility into what the agent is doing (tool calls, agent spawning, draft proposals, verification results) even before the final output arrives. The final response arrives as the `final_response` event.

This is practical because:
- The EventStream already emits 30+ event types during execution
- The daemon already has `stream_events` infrastructure
- No changes needed to the SessionOrchestrator or provider layer
- Users see meaningful progress (not just a spinner)

**Updated `stream_task` method:**
1. Start `executeTask` in background
2. Subscribe to EventStream wildcard for the duration
3. Forward each event as a JSON-RPC notification `{ method: 'task_event', params: { taskId, event } }`
4. On completion, send `{ method: 'task_complete', params: { taskId, result } }`

**Add to types.ts:**
```typescript
export interface StreamTaskParams {
  task: string;
  mode?: string;
  workspaceRoot: string;
}

export interface StreamTaskResult {
  taskId: string;
  status: 'started';
}
```

### Step 3: Add event forwarding in daemon-client

**File:** `chimera/packages/chimera-vscode/src/daemon-client.ts`

Modify `processBuffer()` to handle new notification methods:
- `task_event` → forward to `onTaskEvent` handler
- `task_complete` → forward to `onTaskComplete` handler

Add new handler registration methods:
```typescript
setTaskEventHandler(handler: (taskId: string, event: unknown) => void): void
setTaskCompleteHandler(handler: (taskId: string, result: unknown) => void): void
```

### Step 4: Update chat-panel for streaming display

**File:** `chimera/packages/chimera-vscode/src/chat-panel.ts`

Changes to `processUserInput()`:
1. Call `daemon.call('stream_task', ...)` instead of `execute_task`
2. Add a temporary assistant message placeholder immediately (with "thinking..." content)
3. Register task event handler to receive incremental updates
4. On each `task_event`: update the placeholder message with event summary
5. On `task_complete`: finalize the message with the full output

Changes to `render()` / webview JS:
1. Add a `streaming` class to messages that are still being built
2. Support in-place message content updates without full re-render
3. Add event summary badges (e.g., "Tool: readFile", "Agent spawned: reviewer")
4. Auto-scroll as new content arrives

**Webview message protocol (extension → webview):**
```typescript
{ type: 'stream_start', messageId: string }           // placeholder created
{ type: 'stream_event', messageId: string, event: {} } // intermediate event
{ type: 'stream_end', messageId: string, output: string } // finalize
```

### Step 5: Update webview JavaScript for incremental rendering

**File:** `chimera/packages/chimera-vscode/src/chat-panel.ts` (getHtml method)

Add to the `<script>` block:
- `stream_start` handler: create a new message div with streaming indicator
- `stream_event` handler: append event summary badges to the streaming message
- `stream_end` handler: replace the streaming content with final output, remove streaming class
- Keep `scrollToBottom()` on each update
