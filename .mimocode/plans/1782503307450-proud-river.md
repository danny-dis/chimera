# Concept Comparison: /loop vs /goal vs /workflow vs /skill vs /skill-pack vs /schedule vs /subagent

---

## Quick Reference

| Concept | What It Is | Execution Model | Terminates When | Defined In |
|---------|-----------|-----------------|-----------------|------------|
| **/loop** | Repeat a task N times | Single loop step, count-based | Turn count reached | `registry.ts:258-308` |
| **/goal** | Run until condition met | Single loop step, condition-based | Agent outputs "COMPLETE" | `registry.ts:310-358` |
| **/workflow** | Multi-step DAG of typed steps | Multiple steps (llm/tool/parallel/sequence/gate/loop) | All steps complete or gate fails | `workflow/types.ts` |
| **/skill** | Reusable instruction document | Passive — injected into context as system prompt | N/A (not executed) | `skills/skill-loader.ts` |
| **/skill-pack** | Bundle of skills for a mode | Manifest listing skill names to load together | N/A (not executed) | `skills/skill-pack.ts:34-39` |
| **/schedule** | Cron-based recurring workflow | 60-second timer evaluates cron, dispatches workflow | Manual disable or maxIterations | `workflow/scheduler.ts` |
| **/subagent** | Isolated worker process | V8 worker thread with own context window | Task completes or budget exhausted | `agent/sub-agent-manager.ts` |

---

## Detailed Breakdown

### /loop — Repeat N Times
- **Syntax:** `/loop <turns> <task>` or `chimera loop <turns> <task>`
- **Mechanism:** Creates a `WorkflowDefinition` with a single `loop`-kind step, count-based termination
- **Use when:** Retry a task a fixed number of times

```bash
# REPL
/loop 3 "Run the tests and fix any failures"

# CLI
chimera loop 3 "Run the tests and fix any failures"
```

**Internal workflow it creates:**
```yaml
name: scheduled-loop
steps:
  - id: loop
    kind: loop
    config:
      prompt: "Run the tests and fix any failures"
      max_iterations: 3
      role: writer
```

---

### /goal — Run Until Done
- **Syntax:** `/goal <description>` or `chimera goal <description...>`
- **Mechanism:** Creates a `WorkflowDefinition` with a single `loop`-kind step, `until: 'COMPLETE'`, `max_iterations: 20`
- **Use when:** Keep iterating until a result is achieved, unknown attempt count

```bash
# REPL
/goal "Get all tests passing"

# CLI
chimera goal "Get all tests passing"
```

**Internal workflow it creates:**
```yaml
name: scheduled-loop
steps:
  - id: loop
    kind: loop
    config:
      prompt: "Get all tests passing"
      until: COMPLETE
      max_iterations: 20
      fresh_context: true
      role: writer
```

---

### /workflow — Multi-Step Pipeline
- **Syntax:** `chimera workflow run <name> --input <json>` (CLI only)
- **Mechanism:** Typed DAG with steps of kind: `llm`, `tool`, `parallel`, `sequence`, `gate`, `loop`
- **Use when:** Coordinated multi-phase operations

**Built-in workflows:** `simple-ask`, `simple-plan`, `standard-draft`, `quality-gate`, `parallel-decompose`

```bash
# List available workflows
chimera workflow list

# Show a workflow definition
chimera workflow show standard-draft

# Run a workflow
chimera workflow run standard-draft --input '{"task": "Refactor the auth module"}'
```

**Custom workflow file (`.chimera/workflows/my-pipeline.yaml`):**
```yaml
name: my-pipeline
description: "Draft, review, then merge"
steps:
  - id: draft
    kind: llm
    config:
      prompt: "Write the implementation for: ${task}"
      role: writer

  - id: review
    kind: parallel
    config:
      agents:
        - role: reviewer
          prompt: "Review the code from ${draft.output} for bugs"
        - role: challenger
          prompt: "Challenge assumptions in ${draft.output}"

  - id: gate
    kind: gate
    config:
      condition: "${review.output} contains no critical issues"
      on_fail: abort

  - id: merge
    kind: llm
    config:
      prompt: "Merge feedback: ${draft.output} + ${review.output}"
      role: writer
```

### /skill — Reusable Knowledge (Not Execution)
- **Syntax:** `chimera skill list/show/validate` (CLI only)
- **Mechanism:** Markdown files with YAML frontmatter, loaded into context as system-prompt instructions
- **Use when:** Codify domain knowledge, best practices, procedural instructions

```bash
# List all discoverable skills
chimera skill list

# Show a skill's content
chimera skill show chimera-safety

# Validate a skill file
chimera skill validate .chimera/skills/my-skill.md
```

**Custom skill file (`.chimera/skills/my-skill.md`):**
```markdown
---
name: my-skill
description: "How to handle database migrations safely"
modes: [code, debug]
inputs:
  database: string
  version: string?
---

# Database Migration Guidelines

1. Always backup before migrating
2. Run migrations in a transaction
3. Test rollback before applying
4. Never drop columns without a deprecation period

## Steps
- Check current schema version
- Generate migration SQL
- Apply in transaction
- Verify row counts match
```

**Resolution order:** bundled → `.chimera/skills/` → `~/.config/chimera/skills/` → legacy `.kilo/skills/`

**Bundled skills (9):** `chimera-modes`, `chimera-workflows`, `chimera-safety`, `chimera-tool-loop`, `chimera-cli`, `chimera-telemetry`, `chimera-cost-control`, `chimera-learning`, `chimera-skill-creation`

### /skill-pack — Bundle of Skills for a Mode
- **File location:** `.chimera/skill-packs/<mode>.md`
- **Mechanism:** YAML frontmatter manifest listing skill names to load together for a specific agent mode
- **Use when:** You frequently use the same组合 of skills together and want them auto-loaded

```markdown
---
name: security-audit-pack
description: "Skills for security auditing tasks"
mode: code
skills:
  - chimera-safety
  - chimera-tool-loop
  - chimera-telemetry
---
```

**Another example — debug pack:**
```markdown
---
name: debug-pack
description: "Full debugging toolkit"
mode: debug
skills:
  - chimera-tool-loop
  - chimera-telemetry
  - chimera-cost-control
  - chimera-safety
---
```

**How it works:**
1. When the agent enters `code` mode, `loadSkillsForMode('code')` runs
2. First loads static bundle skills (bundled)
3. Then loads `.chimera/skill-packs/code.md` — pack skills **override** bundle skills with the same name
4. Then loads workspace skills from `.chimera/skills/*.md` matching the mode

**Auto-synthesis:** The learning engine analyzes skill co-occurrence across sessions and auto-generates skill packs when it finds patterns (e.g., if you always use `chimera-safety` + `chimera-tool-loop` together in debug mode, it creates a pack).

### /schedule — Cron-Based Recurring Workflow
- **File:** `.chimera/schedules.json`
- **Mechanism:** Cron expression triggers workflow dispatch every 60 seconds
- **Use when:** Automate recurring tasks — nightly builds, periodic reviews, scheduled reports

**Schedule entry format (`.chimera/schedules.json`):**
```json
[
  {
    "id": "sch-abc123",
    "workflow": "standard-draft",
    "task": "Review all open PRs and summarize findings",
    "cron": "0 9 * * 1-5",
    "inputs": {},
    "enabled": true,
    "createdAt": "2026-06-24T00:00:00Z",
    "maxIterations": 10
  },
  {
    "id": "sch-def456",
    "workflow": "quality-gate",
    "task": "Run full test suite and report coverage",
    "cron": "0 2 * * *",
    "inputs": {},
    "enabled": true,
    "createdAt": "2026-06-24T00:00:00Z"
  }
]
```

**Cron format (5-field):**
```
┌────── minute (0-59)
│ ┌────── hour (0-23)
│ │ ┌────── day of month (1-31)
│ │ │ ┌────── month (1-12)
│ │ │ │ ┌────── day of week (0-6, Sun=0)
│ │ │ │ │
* * * * *
```

**Common patterns:**
| Cron | Meaning |
|------|---------|
| `0 9 * * 1-5` | Weekdays at 9:00 AM |
| `0 2 * * *` | Daily at 2:00 AM |
| `*/15 * * * *` | Every 15 minutes |
| `0 0 1 * *` | First of each month |

**How it works:**
1. `SchedulerManager` starts a 60-second interval timer
2. Every tick, `evaluate()` checks all enabled schedules against current time
3. Double-trigger guard prevents same-minute re-runs
4. Matching schedules dispatch their workflow via `WorkflowDispatcher`
5. If no workflow defined, builds a default loop workflow from the `task` field

### /subagent — Isolated Worker Process
- **Syntax:** `chimera parallel <task>` (CLI). Internal: `SubAgentManager.spawnSubAgent()`
- **Mechanism:** Real process-isolated execution unit in its own V8 worker thread
- **Use when:** Fan out computation to independent workers

```bash
# CLI — parallel execution
chimera parallel "Review module A" & chimera parallel "Review module B"
```

**Internal usage (from a workflow or agent):**
```typescript
// Spawn a single subagent
const handle = await subAgentManager.spawnSubAgent({
  task: "Review the auth module for security issues",
  budget: 0.50,  // $0.50 USD cap
});

const result = await handle.result;

// Spawn multiple in parallel
const handles = await subAgentManager.spawnParallel([
  { task: "Review module A", budget: 0.25 },
  { task: "Review module B", budget: 0.25 },
  { task: "Review module C", budget: 0.25 },
]);

const results = await Promise.all(handles.map(h => h.result));
```

**Key characteristics:**
- Each subagent gets its own context window, provider connection, and budget cap
- Default budget: $1.00 USD per subagent
- Emits `subagent_spawned` and `subagent_completed` events
- Used by `parallel-decompose` workflow to fan out N workers

## Composition Relationships

```
/skill-pack  ←── bundles ──  /skill (N skills per mode)
/schedule    ←── dispatches ──  /workflow (cron triggers execution)
/workflow    ←── spawns ──  /subagent (parallel steps fan out workers)
/workflow    ←── generalized by ──  /goal and /loop (single loop-step workflows)
/skill       ←── injected into ──  /workflow (llm steps) and /subagent context
```

**Resolution hierarchy for skill loading:**
```
SKILL_BUNDLES[mode]  (static, bundled)
       ↓ override
.chimera/skill-packs/<mode>.md  (pack, per-mode)
       ↓ supplement
.chimera/skills/*.md  (workspace, per-mode)
```

## When to Use What

| Scenario | Use |
|----------|-----|
| "Try this N times" | `/loop` |
| "Keep going until it works" | `/goal` |
| "Do A, then B in parallel with C, then merge" | `/workflow` |
| "Teach the agent how to do X properly" | `/skill` |
| "I always use these 3 skills together in debug mode" | `/skill-pack` |
| "Run this review every weekday at 9am" | `/schedule` |
| "Run multiple independent workers concurrently" | `/subagent` (via workflow `parallel` step or `chimera parallel`) |
